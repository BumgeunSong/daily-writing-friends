## Change Retro: integrate-signup-cohort-flow

### Timeline

- **2026-05-05 — Proposal & design.** Two-phase author session produced `proposal.md` and `design.md` already in the worktree before this implementation cycle. Reviewers caught four critical and several important findings; the author folded them into the design before handoff.
- **2026-05-05 — Implementation cycle starts.** 68-task `tasks.md` opened. Section 0 spike was the first hard gate.
- **2026-05-05 — Spike against local Supabase.** Four findings reshaped the design mid-flight: `signUp` returns 422 for already-registered emails (no email queued), `verifyOtp` is stateless (no session binding), Supabase merges `expired` and `invalid_token` into one `otp_expired` code, and the auto-link path the design assumed does not actually link a password identity. Documented in `docs/plans/2026-05-05-otp-spike-report.md`.
- **2026-05-05 — Apply phase.** Sections 1–14 implemented in dependency order. The spike narrowed scope: the `success-linked` UI state (task 11.3) and the auto-link e2e (task T.9) were dropped, and `SignupPage` rerouted Persona B to `/login` instead of `/verify-email`.
- **2026-05-05 — Verify (round 1).** `tsc` clean; vitest 709/709; two SQL fixture tests passed against local Postgres. Live UI flow run through `agent-browser`: signup → Mailpit OTP → onboarding → save (phone tab + kakao tab variants), wrong-OTP error state, Persona B inline error, and dispatcher routing all matched the spec.
- **2026-05-05 — Self-requested code review.** Five parallel reviewers (general, security, silent-failure, test-coverage, quality). Three Critical and four Important findings. Pushback on one false-positive ("dead branch" was actually intentional per design D8). Five follow-up commits resolved the rest, +6 unit tests added.
- **2026-05-05 — Spec alignment.** Backfill scenarios in the spec were narrowed to match the migration's defensive `AND contact present` guard. All ten requirements aligned.
- **2026-05-06 — PR opened (#571).** Initial CI run failed on `e2e` due to a migration timestamp collision: `20260505000000_donations.sql` had been merged to `main` while this branch was open. Fixed by renaming the file to `20260506000000_…`, merging `origin/main` into the branch, and re-running locally (test count rose 709 → 723 from the merged donator suite).
- **2026-05-06 — CI green on round 2.** All 8 active checks passed, including the e2e job that exercises the full migration set.

### Wins

- **Spike-first, scope-second.** The 60-minute Section 0 spike caught three load-bearing wrong assumptions before any UI code. Without it, Persona B would have shipped as a screen that waits for an OTP that never arrives.
- **Pure-function seam.** `resolveOnboardingSubmit`, `decideVerifySuccessState`, `validatePhone/validateKakaoId`, `classifySupabaseAuthError` test cleanly without React Testing Library or Supabase mocks. Six of those tests caught fragile edges during apply, not in CI.
- **Five-reviewer fan-out.** Running general / security / silent-failure / test / quality in parallel surfaced findings each reviewer alone would have missed (the silent-failure-hunter caught the `useOnboardingComplete` swallowed-error path; the quality-reviewer caught the regex collision risk in `classifySupabaseAuthError`). Cost was minutes; payoff was three Critical fixes before push.
- **Defensive backfill.** The migration's `AND contact present` clause caught the spec's implicit `permission ⇒ contact` invariant. The first SQL fixture test would have crashed the migration without it.
- **agent-browser end-to-end run.** Catching the empty `/boards` state, the toast persistence, and the Persona B inline copy at the same screen the user sees made the verify report trustworthy in a way unit tests cannot match.

### Misses

- **Migration timestamp collision (Tool Gap, surfaced in CI).** Local `main` was 123 commits stale, hiding the parallel `20260505000000_donations.sql` merge. CI was the first place the collision showed up. **Root cause:** the apply phase committed the migration before fetching latest `origin/main`. **Cost:** one CI round, one rename + merge.
- **`JoinDispatcher` apparent-dead-branch on initial review (Process Gap).** Two reviewers flagged `if (!onboardingComplete) navigate(ONBOARDING) ... navigate(ONBOARDING)` as a bug; it was actually intentional (design D8 sends "onboarded but inactive non-waitlist" users to OnboardingPage as their re-apply surface). The code looked wrong even though the destination was right. **Cost:** review noise; pushback was correct but a `void onboardingComplete` plus a comment would have prevented the misread up front.
- **`updateUser` order risk caught in review, not in apply (Review Gap, then closed in review).** Original apply wrote `onboarding_complete=true` before the waitlist insert. Quality-reviewer flagged the partial-failure case. Reordered to `profile → waitlist → onboardingComplete=true` so any crash leaves the user on a recoverable retry path.
- **Pre-fill error swallowed silently in apply (Review Gap, then closed in review).** Silent-failure-hunter caught `OnboardingPage`'s pre-fill `try/finally` that proceeded to a blank form on `fetchUser` error, allowing accidental NULL overwrite. Added a `prefillError` state, alert, and submit-disabled guard. The review caught what the test plan missed.
- **`SignupPage` 422 path had no automated test (Test Gap, closed in review).** The spike-driven UX (Persona B inline error) shipped without an RTL test. Pr-test-analyzer flagged it; the missing test was added in the review-fix commits.
- **`tasks.md` lagged the migration rename (small process drift).** Task 2.1 still named the old timestamp until `final-spec-alignment.md` caught it. Updated in the same commit.

### Improvement Ideas

- **Pre-PR migration-collision check.** A trivial bash one-liner that, before opening a PR, fetches `origin/<base>` and `comm -12` the migration filenames in `HEAD..origin/<base>` against `origin/<base>..HEAD` would catch any timestamp collision in seconds. Could live as a `pre-push` hook or as part of the project's `npm run validate`. Skill `verify-runtime` already gestures at this for runtime drift; add a "merge surface check" sibling.
- **Mark-design-collapsed-branches explicitly.** When two semantic predicates legitimately produce the same destination (D8 in this change), the convention should be a `// design D8: collapse intentional` comment plus a `void` reference to the unused predicate. Cheap insurance against the "looks like a bug" review noise. Document it in the project's CLAUDE.md so it becomes house style.
- **Bake "Persona B test fixture" into `authErrors`.** The 422 `user_already_exists` case is now load-bearing in three places (Sentry classification, signup branching, error message). A single fixture object reused by `isAlreadyRegisteredError` test, `SignupPage.test.tsx`, and any future caller would prevent the next Supabase upgrade from silently changing the error shape.
- **Expand the OTP spike checklist into a runnable script.** The spike was manual curl + Mailpit + psql. Capturing it in `scripts/spike-otp.ts` would let the next person re-run it in one command when Supabase publishes a CLI / SDK upgrade. Tracking script + report path in the OpenSpec change template would normalize "spike before UI rewrite" as a first-class step.
- **Per-email OTP rate-limit follow-up.** D11 deferred this with an explicit follow-up plan (Edge Function wrapper). Track it in a separate OpenSpec change so the security trade-off doesn't decay silently.
- **Re-evaluate `useOnboardingComplete` "false on error".** The current contract is "default to onboarding form on error" with a console.error. If onboarding-complete users start seeing transient redirects to `/join/onboarding` in production logs, switch to a third-state ("unknown") UI rather than the silent-default we have now. The current trade-off is correct given the cost of a feature flag, but worth revisiting after deploy data lands.
